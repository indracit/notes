
### 1. The Core Conceptual Model: Buckets & Objects

**Objects:**

 The _actual piece of data_ you are storing. This could be a text file, a photo, a video, a database backup, or even an entire application.

An object consists of:

- **Key (Name):** The unique identifier for the object within a bucket (e.g., `images/logo.png`).

 - **Value (Data):** The byte-stream of the content itself.
 
- **Metadata:** A set of name-value pairs (e.g., `content-type: image/png`, `last-modified: 2024-05-20`). Some are system-defined, and you can add custom ones.

-  **Version ID:** (If versioning is enabled) a unique ID for that specific version of the object.

- **Access Control List (ACL):** (Legacy) Defines who can access the object.

**Buckets:** 

The _containers_ that hold objects. Think of a bucket as a top-level storage unit, analogous to a root directory on a drive, but with specific properties.

- **Globally Unique Name:** The bucket's name must be globally unique across _all_ of AWS, not just your account. (e.g., `my-company-logs-2024` is good, `logs` is probably taken).

-  **Region-Specific:** A bucket is created in a specific AWS Region (e.g., `us-east-1`, `eu-west-2`). Data never leaves that region unless you explicitly configure it to.

- **Flat Structure:** Despite the UI showing "folders," buckets have a flat structure. The key `project/report.pdf` is just an object with a very long name.

**Key Insight:** No folders. Just a flat map of **`Bucket Name + Object Key`** pointing to data. The "/" in keys is purely a visual convenience.


### 2. The Architectural Promise: Durability & Availability

 - **Durability (Data Persistence):** 
	S3 is designed for **11 nines of durability** (99.999999999%). By automatically replicating and distributing your object across **multiple Availability Zones (AZs)** within the chosen region. A failure of one entire data center does not cause data loss. This is automatic and transparent.
	
- **Availability (Uptime for Requests):**
	This is different from durability. A service can be durable (your file is safe somewhere) but briefly unavailable (you can't fetch it right now). S3 Standard offers **99.99% availability**, meaning it accepts and processes requests almost all the time. Other storage classes trade availability for lower cost.

- **Strong Read-After-Write Consistency for PUTS and DELETES:** 
	If you upload (`PUT`) a _new_ object, a subsequent `GET` or `LIST` request will immediately see it.

### 3. The Storage Classes

**Theoretical Takeaway:** Choose a class not just by "how often," but by your tolerance for retrieval latency and availability risk.

### 4. Security Model

- **Identity-based Policies (IAM):** You attach a policy to an IAM User, Group, or Role. "Allow `s3:GetObject` on `my-bucket` for `user@company.com`." This answers _who_ has permissions.

- **Resource-based Policies (Bucket Policies):** You attach a JSON policy _directly to the bucket_. This is powerful for granting cross-account access or making a bucket public. "Allow `s3:GetObject` for `Principal: *` (the public internet)." This answers _what_ permissions are on the resource.

- **Access Control Lists (ACLs) - Legacy:** Older, finer-grained permission system for individual objects. Best practice is to use IAM and Bucket Policies instead.

- **Pre-signed URLs:** A core theoretical concept for temporary access. You generate a URL that grants limited access (e.g., `PUT` or `GET` on a specific object) for a specific time (e.g., 5 minutes).

- **Block Public Access:** A set of "override" settings that act as a safety net. Even if a bucket policy tries to make data public, these settings will block it.

### 5. Data Management Features

**Lifecycle Policies:** 

A set of rules to automatically transition objects between storage classes or expire (delete) them.

- _Example Rule:_ "After 30 days, move objects from S3 Standard to S3 Standard-IA. After 90 days, move to S3 Glacier Deep Archive. After 365 days, delete them."

**Versioning:** A feature at the bucket level. When enabled, instead of overwriting an object, S3 retains the old version and gives it a new `Version ID`. The current object has a `null` version ID.

`GET` without a version ID gets the latest version. `DELETE` creates a "delete marker" (the object becomes invisible but can be restored by deleting the marker).

**Replication (CRR & SRR):**

- **Cross-Region Replication (CRR):** Automatically copy objects to a bucket in a _different_ AWS region. Use cases: lower latency for global users, compliance requirements, disaster recovery.

- **Same-Region Replication (SRR):** Copy objects to another bucket in the _same_ region. Use cases: aggregating logs into a central bucket, replicating data for production/test environments.

### 6. Performance

- **Baseline Performance:** 
  S3 can achieve **at least 3,500 PUT/COPY/POST/DELETE requests per second** and **5,500 GET/HEAD requests per second** _per partitioned prefix in a bucket_.

 - **Prefixes:** A prefix is the object key up to the last "/". `images/2024/thumbs/` is a prefix.
- **The Scaling Trick:** To get higher throughput, spread your requests across many prefixes.
    
- `bucket/customers/1/data.json`    
- `bucket/customers/2/data.json`
- Using the above, you can achieve 5,500 GETs for `customers/1/` **AND** 5,500 GETs for `customers/2/` **simultaneously**.

- **Multipart Upload:** For large objects (>100MB), you should upload it in parallel parts. This gives higher throughput, faster recovery from network errors, and the ability to pause/resume.
